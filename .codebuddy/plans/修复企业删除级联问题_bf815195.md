## 问题描述

用户反馈：在首页删除企业后，该企业的企业档案、数据管理没有对应删除，并且报告管理等页面还有被删除企业的信息。

## 问题分析

`CompanyService.deleteCompany()` 方法只级联删除了联系人（`contact`）和企业主表（`company`），缺少以下关联数据的级联删除：

| 关联数据 | 表名 | 当前状态 |
| --- | --- | --- |
| 数据文件 | `data_file` | 未删除 |
| 企业子模板 | `company_template` | 未删除 |
| 占位符状态 | `company_template_placeholder` | 未删除 |
| 报告 | `report` | 未删除 |


## 修复目标

1. 在 `CompanyService.deleteCompany()` 中添加完整的级联删除逻辑
2. 在各关联 Mapper 中添加 `deleteByCompanyId` 方法（逻辑删除）
3. 确保删除顺序正确（先删子表，再删主表）
4. 所有操作在事务中执行，保证原子性

## 技术方案

### 实现策略

采用逻辑删除（`deleted = 1`）而非物理删除，保留数据可追溯性。利用 MyBatis-Plus 的 `BaseMapper` 能力，通过 `update` 语句设置 `deleted = 1`。

### 删除顺序

```
1. 删除数据文件 (data_file)
2. 删除占位符状态 (company_template_placeholder) - 需先查询该企业所有子模板ID
3. 删除企业子模板 (company_template)
4. 删除报告 (report)
5. 删除联系人 (contact)
6. 删除企业主表 (company)
```

### 关键实现点

1. **占位符状态删除**：`company_template_placeholder` 表没有 `company_id` 字段，需先查询该企业所有子模板ID，再批量删除占位符
2. **文件清理**：逻辑删除后，文件仍保留在磁盘，如需物理删除文件需额外处理
3. **租户隔离**：所有删除操作需带上 `tenant_id` 条件，防止跨租户数据操作
