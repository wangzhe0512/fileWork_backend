## 产品概述

修复反向生成子模板时的行类型判断逻辑，使采购/销售明细表等动态行数表格能正确生成占位符标记。

## 核心问题

从截图分析，当前生成的子模板存在以下问题：

1. 采购明细表和销售明细表的所有数据行都被错误标记为 `{{_row_group}}`，而非正确的 `{{_row_data}}`
2. 末行合并单元格（合计行）被错误标记为普通data行，缺少 `{{_row_subtotal}}` 标记
3. 所有行缺少 `{{_tpl_...}}` 前缀（只有data行才应该写tplMark）

## 根本原因

当前行类型判断逻辑（ReverseTemplateEngine.java 1746-1753行）：

```java
// group判断：列数 == 表头列数 && 第0列非空
if (rowCells.size() == headerColCount && !col0Str.isEmpty()) {
    rowTypeMark = "{{_row_group}}";
} else {
    rowTypeMark = "{{_row_data}}";
}
```

这个逻辑错误地把所有普通数据行（3列，col0非空）都判断成了group行。

## 正确的行类型特征

- **data行（明细数据行）**：单元格数 = 表头列数，每列都有实际数据，标记为 `{{_tpl_...}} {{_row_data}} {{_col_字段名}}`
- **group行（分组标题行）**：该行只有1个单元格且跨满整行（gridSpan = 表头列数），标记为 `{{_row_group}} {{_col_字段名}}`
- **subtotal行（小计/合计行）**：任意列含"小计"/"合计"/"总计"，或末行整行合并，标记为 `{{_row_subtotal}} {{_col_字段名}}`

## 技术栈

- Java 17
- Apache POI 5.x（XWPF操作Word文档）
- XMLBeans（底层XML操作）

## 实现方案

### 核心修改

修改 `ReverseTemplateEngine.java` 中 `TABLE_ROW_TEMPLATE` 分支的行类型判断逻辑：

1. **新增辅助方法**：判断单元格是否有跨列合并（通过 `CTTcPr` 获取 `gridSpan` 值）
2. **重写行类型判断逻辑**：

- 优先判断 `subtotal`：文本含"小计/合计/总计" 或 末行且整行合并
- 判断 `group`：整行合并（gridSpan之和 = 表头列数）且不是末行
- 其余为 `data` 行
